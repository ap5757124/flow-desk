# 面试亮点

本文件记录技术面试中应该容易讲清楚的项目亮点。

## 1. 模块化单体

FlowDesk 没有一开始就使用微服务，而是采用模块化单体。

面试讲法：

> 我选择模块化单体，是因为项目更容易运行、测试和理解，同时仍然保持清晰的模块边界。如果业务继续增长，notification、file storage、reporting 等模块后续可以再拆分成独立服务。

## 2. RBAC 加数据权限

RBAC 控制用户能做什么，数据权限控制用户能访问哪些数据。

面试讲法：

> 我把功能权限和数据权限拆开设计。用户可能有查看工单的权限，但只能查看自己租户、部门或分派范围内的工单。

## 3. 工单状态机

工单状态流转由明确规则控制。

示例：

```text
NEW -> ASSIGNED -> PROCESSING -> WAITING_CONFIRM -> CLOSED
PROCESSING -> SUSPENDED -> PROCESSING
WAITING_CONFIRM -> PROCESSING
NEW -> CANCELLED
```

面试讲法：

> 我没有把 if-else 状态判断散落在各个 Service 中，而是集中维护工单流转规则。这样可以统一校验源状态、目标状态、操作角色和副作用。

## 4. Outbox Pattern

业务数据和事件记录在同一个数据库事务中保存。

面试讲法：

> 创建工单时，我不会在事务里直接依赖 MQ 发送成功，而是把工单记录和 Outbox 事件一起落库。后续由发布器投递到 MQ 并支持失败重试，这样可以避免 MQ 临时不可用导致通知丢失。

## 5. 幂等设计

重要的重复操作不能产生脏数据。

示例：

- 重复创建工单请求
- 重复点击催办
- MQ 重复消费
- 重复状态流转请求

面试讲法：

> 对用户操作，我可以使用 requestId 或业务唯一键；对 MQ 消费者，我使用 eventId 和消费记录，保证重复投递不会造成重复副作用。

## 6. 乐观锁

工单更新使用 `version` 字段防止并发状态覆盖。

面试讲法：

> 如果两个处理人同时操作同一个工单，第一次更新成功并递增 version。第二次更新会因为 version 不匹配而失败，用户需要刷新后重试。

## 7. 业务时间线

每个重要工单操作都记录为时间线事件。

面试讲法：

> 时间线不仅用于前端展示，也支持审计、问题排查、客户沟通历史和 SLA 复盘。

## 8. 多租户隔离

业务表使用 `tenant_id`。

面试讲法：

> 租户上下文来自认证用户。业务查询自动应用租户过滤，文件下载也会校验租户归属，避免跨租户数据泄露。

## 9. 异步通知

通知由领域事件驱动。

面试讲法：

> ticket 模块只产生事件，notification 模块消费事件并发送站内信、邮件或其他渠道通知。这样降低了工单处理和通知投递之间的耦合。

## 10. 查询和报表优化

运营查询和统计报表采用不同策略。

面试讲法：

> 工单列表围绕 status、assignee、creator、tenant、created_at 设计索引。报表可以使用定时预聚合，避免每次看板请求都扫描大业务表。

# 架构设计

## 架构风格

默认使用模块化单体架构。

这样既能保持项目真实、可维护，又不会过早引入分布式系统复杂度。微服务不是第一版的必要选择，否则会增加部署、链路追踪、服务治理和分布式一致性成本，反而干扰面试项目的核心目标。

## 逻辑分层

推荐分层：

- `controller`：HTTP 入口、触发请求校验、响应映射
- `application`：用例编排、事务边界
- `domain`：实体、领域服务、状态规则、业务不变量
- `repository`：数据库访问接口和 mapper
- `infrastructure`：Redis、MQ、MinIO、搜索、外部系统适配器

## 推荐包结构

```text
com.example.flowdesk
├── audit           # 操作日志、审计、时间线
├── common          # 通用能力
├── file            # 文件、附件、MinIO 
├── infra           # 基础设施适配，比如 Redis、MQ、对象存储
├── notification    # 通知、站内信、MQ 消费
├── report          # 统计报表
├── security        # 登录认证、权限、安全上下文
├── system          # 用户、角色、部门、菜单、字典
├── tenant          # 租户、多租户隔离
├── ticket          # 工单核心业务
└── workflow        # 状态机、流程流转
```



每个业务模块可以使用类似结构：

```text
ticket
├── controller       # 控制层
├── service          # 业务接口
│   └── impl         # 业务接口实现
├── mapper           # MyBatis / MyBatis-Plus mapper
├── entity           # 数据库实体
├── dto              # 数据传输对象
│   ├── request      # 接口接收参数
│   └── response     # 接口返回参数
├── enums            # 枚举
├── convert          # 对象转换
└── domain           # 复杂业务规则，可选
```

## 事务规则

- 事务边界放在 application service。
- 事务尽量短。
- 不要在数据库事务中调用慢外部服务。
- 如果某个动作需要 MQ 或外部通知，在同一事务中写入事件记录，再异步发布。

## 权限模型

使用两层权限：

- 功能权限：用户是否可以调用某个 API 或操作某个菜单。
- 数据权限：用户是否可以查看或修改某一行数据。

数据范围示例：

- 本人数据
- 本部门数据
- 本部门及下级部门数据
- 当前租户全部数据
- 平台管理员全局数据

## 多租户模型

业务表使用 `tenant_id`。

规则：

- 所有租户业务查询都必须包含租户过滤。
- 租户上下文来自已认证用户上下文。
- 管理员操作必须明确是租户级还是平台级。
- 文件访问、工单访问、日志和统计都必须遵守租户范围。

## 异常处理

使用统一业务异常和错误码。

推荐错误分类：

- 参数校验错误
- 认证错误
- 授权错误
- 数据不存在
- 业务状态冲突
- 重复操作
- 外部依赖失败
- 系统错误

## 可观测性

建议基础能力：

- 请求 traceId
- 重要业务动作结构化日志
- 慢 SQL 观察
- MQ 消费日志
- 带业务上下文的错误日志
- 可选 Micrometer 指标

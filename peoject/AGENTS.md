# AGENTS.md

本仓库用于 FlowDesk，一个面向后端技术面试的 Spring Boot 项目。

FlowDesk 是一个企业级工单协同平台，类似 IT 服务台或企业内部流程系统。项目目标不是堆砌功能，而是展示真实后端工程能力：分层架构、业务闭环、安全认证、权限控制、事务设计、异步处理、可观测性、测试，以及能在面试中讲清楚的技术决策。

## 项目目标

本项目不是简单 CRUD 示例。每个功能都应该按照真实企业后端功能来设计和实现。

优先关注：

- 清晰的模块边界
- 从请求到落库的完整业务链路
- 企业级编码规范
- 安全、权限和数据访问控制
- 事务、幂等和一致性设计
- 可测试、可解释的实现
- 能帮助后续 AI 会话继续协作的项目文档

## 产品方向

FlowDesk 是一个企业级工单协同平台。

核心业务范围：

- 用户认证与授权
- RBAC 权限
- 组织和部门管理
- 工单创建、分派、处理、确认、关闭和评价
- 工单评论和协作时间线
- 附件上传和带权限控制的下载
- 操作日志和审计追踪
- 通知和消息投递
- 统计和报表
- 多租户数据隔离
- 基于状态机的工单流转
- 异步领域事件和基于 Outbox 的最终一致性

## 推荐技术栈

## AI 协作规则

每次新的 AI 会话必须先阅读：

1. `AGENTS.md`
2. `docs/00-project-vision.md`
3. `docs/01-architecture.md`
4. `docs/features/` 下和当前任务相关的文件
5. `docs/adr/` 下和当前任务相关的架构决策记录

实现任何功能前，除非用户明确要求直接编码，否则 AI 必须先输出简短设计方案并等待确认。

每个功能设计至少覆盖：

1. 业务目标
2. 请求入口
3. 权限和数据范围校验
4. 核心业务规则
5. 数据库表和关键字段
6. 事务边界
7. 异常处理
8. 审计和操作日志
9. 是否需要异步事件
10. 测试策略
11. 面试讲解点

实现完成后，AI 应该：

- 说明改了什么
- 列出影响到的模块
- 当行为或架构发生变化时更新相关文档
- 给出验证方式
- 避免无关重构

## 架构原则

- Controller 只处理 HTTP 相关职责。
- Application service 负责编排用例和事务边界。
- Domain service 放置不属于单一实体的业务规则。
- Infrastructure adapter 隔离 Redis、MQ、对象存储、搜索和外部系统。
- 不要把业务逻辑直接写在 Controller 中。
- 不要让一个 Service 类承载过多混杂职责。
- 优先使用明确的 request DTO、response DTO 和 command 对象。
- 重要业务状态使用 enum 和状态机表达。
- 有意识地设计数据库约束和索引。
- 重要业务动作要写业务日志和审计记录。

## 功能完成标准

一个功能只有满足以下条件，才算完成：

- API 契约清晰
- 输入校验完整
- 权限校验明确
- 数据范围校验生效
- 核心业务实现完成
- 持久化和事务处理正确
- 异常处理明确
- 必要时记录操作日志或审计时间线
- 有测试或记录清楚的手工验证方式
- 重要行为已同步更新文档

## 禁止事项

- 不要开发没有业务意义的纯 CRUD 功能。
- 不要为了展示技术而引入没有必要的技术。
- 不要跳过权限、异常、事务或日志设计。
- 不要把所有逻辑堆进 Controller 或一个巨大的 Service。
- 涉及业务数据的功能不能忽略租户隔离。
- 没有 ADR 时，不要破坏已有架构约定。
- 除非用户明确要求，不要覆盖用户已有修改。
